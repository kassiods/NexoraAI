Nexora - visão geral da aplicação (fev/2026)
=================================================

Público e proposta
- Comunidade focada em hubs de conhecimento, feedback rápido e networking.
- Experiência dark theme com PWA (manifest e sw.js).

Arquitetura (Next.js 14 / App Router)
- Pastas: src/app/(auth) para fluxos públicos; src/app/(app) para áreas logadas; src/app/(app)/admin para painel admin.
- Componentes compartilhados em src/components (dashboard, layout, ui).
- Dados e serviços mockados em src/data/mock/* e src/services/* (Firebase/Firestore simulados por mock-backend).
- Tailwind para UI; tokens de cor/raio/sombra em src/app/globals.css.

Autenticação e perfil
- Hooks: use-auth, use-require-profile garantem usuário carregado e perfil completo.
- Fluxos: login, registro, reset de senha (mock), completar perfil (username fixo com @, displayName, área).
- AuthGateStatus mostra login/sair no topo quando usado.

Shell e navegação
- AppShell: sidebar + mobile nav com links Inicio, Hubs, Notificações, Perfil (Admin aparece se role=admin).
- Mobile: menu inferior fixo; backdrop ao abrir drawer.

Tela Inicio / Dashboard (src/app/(app)/dashboard)
- Composer (placeholder) para compartilhar progresso (mensagem mockada).
- Feed com ActivityCard: mostra autor, hub, tempo, conteúdo, likes com contador, comentar toggle, denunciar.
- Likes: estado local por post; contador refletindo array de likes de cada post.
- EmptyFeedState sugere hubs quando feed vazio.
- Cards "Seu contexto" usam StatCard (hubs do usuário, posts recentes, status do perfil).

Hubs
- Listagem (src/app/(app)/hubs): filtro por categoria, busca com debounce, CTA "Solicitar novo hub". Cards de hub mostram categoria, status (aberto/por solicitação) e CTA.
- Detalhe do hub (src/app/(app)/hubs/[id]):
  - Postar no feed do hub (texto livre). Novo post já grava like do autor e contador inicia em 1.
  - Posts listam autor, tempo relativo, conteúdo em branco alto contraste, like com contador, comentar, denunciar.
  - Comentários por post: input inline, lista com autor, tempo, conteúdo; likes de comentário têm contador local.
  - ReportModal usado para denúncias.
  - Paginação simples via "Carregar mais".
- Solicitar hub (src/app/(app)/hubs/requests/new): formulário com validação Zod (nome, descrição, categoria, objetivo opcional). Envia mock para hubService, mostra sucesso/erro amigável.

Notificações
- Página src/app/(app)/notifications/page.tsx (mock data) lista notificações com filtros básicos (ler/não lida) e CTA para hubs/perfil.

Perfil
- Página src/app/(app)/profile e /profile/[username]: mostra dados mock, estatísticas, badges; CTAs de editar (mock) e copiar perfil.

Admin
- src/app/(app)/admin: atalho.
- src/app/(app)/admin/reports: painel de moderação.
  - Lista denúncias pendentes e resolvidas (mockServices: moderationService, mockPosts/comments/hubs/users).
  - Para pendentes: selecionar ação (nenhuma, restrição temporária de posts/comentários, banimento). Valida nota/duração. Confirmação via window.confirm. Atualiza status no estado local.
  - Mostra contexto: conteúdo, autor, hub (link tipado), histórico mock do usuário.

Serviços mock
- postService: lista feed por hubs, cria post, addComment. Likes são arrays de user IDs no Post.
- hubService: lista hubs, categorias, solicita hubs.
- moderationService: recebe reportContent, resolveReport (mock).
- userService: signIn/signUp/updateProfile (mock backend), getById.

UI e microinterações
- Buttons/input utilitários em globals.css (.btn, .input-field, .transition-base, .hover-lift). Focus-visible com halo.
- Transições 150–180ms, easing padrão.
- Textos de estado (erros/sucessos) são curtos e não técnicos.
- Cores: base/surface escuros, texto primário/secondary, action clara; text-contrast e usos white/90 para conteúdo.

PWA
- manifest.json aponta para /icons/icon-192.svg e /icons/icon-512.svg.
- sw.js e workbox já presentes; next-pwa registra SW.

Acessibilidade
- aria-busy em botões de submit; aria-live em mensagens de status; aria-current em navegação.

Pontos de atenção
- Dados são mocks (não há persistência real).
- Likes/comentários funcionam no estado do cliente; recarregar perde estado.
- Composer do dashboard é placeholder (não cria post).

Principais fluxos resumidos
1) Entrar/Registrar → se novo, completar perfil → redireciona para Inicio.
2) Inicio: ver feed dos hubs do usuário; curtir/denunciar; seguir para hubs.
3) Hubs: descobrir filtros; abrir hub; postar/curtir/comentar/denunciar; carregar mais.
4) Solicitar hub: envia pedido e mostra confirmação.
5) Admin: revisar denúncias, aplicar ação e marcar como resolvida.

Fluxos de interação (detalhe)
- Login: SignInForm -> userService.signIn -> redirect /dashboard (ou redirect query). Erro amigável.
- Registro: SignUpForm -> userService.signUp -> updateProfile (username/displayName null) -> /auth/complete-profile.
- Completar perfil: grava username/displayName/area -> /dashboard.
- Dashboard feed: ActivityCard recebe post.likes; onLike atualiza estado local de liked; contador exibe length. Comentar toggle (draft local, não publica real). Denunciar abre ReportModal (reportTarget em dashboard state).
- Hubs detalhe: postService.listByHub carrega posts/comments; likes de post com contador; comentários têm contador separado; postar cria post e já registra like do autor; denunciar seta reportTarget e envia via moderationService.reportContent.
- Solicitar hub: validação Zod; envia requestHub; mostra alerta sucesso/erro; mantêm category input dual (select + custom).
- Admin reports: carrega moderationService.listReports; permite marcar ação (none/moderate-comments/moderate-posts/severe-ban), validar nota/dias, confirmar; resolveReport atualiza estado; mostra contexto do conteúdo e links para hub/perfil.
- Notificações: lista mock notifications; filtros básicos (ler/não lida) no componente (simples).
- Perfil: apresenta dados mock; CTAs placeholders (copiar, editar). Perfil público lê por username.

Limitações conhecidas
- Persistência inexistente: tudo em memória mock-backend; recarregar perde likes/comentários novos.
- Composer da home não publica posts (mensagem informativa).
- Middleware desativado: rotas não exigem auth real.
